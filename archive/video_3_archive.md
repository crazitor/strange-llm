## 内容大纲

### 1. 开场与部署简介（00:00 - 01:00）
应观众要求加更一期OpenClaw内容。快速介绍部署方案：腾讯云轻量服务器 + OpenRouter大模型 + 飞书机器人。配置流程包括购买服务器、配置API Key、在飞书开放平台新建机器人并配置回调。强调与普通Agent无本质区别，最大不同是可接入社交媒体。

### 2. 简单任务实测：消息处理全过程（01:00 - 02:26）
以"你有多少内存"为例，完整展示消息处理链路：user消息 -> assistant分析（识别出是问服务器内存）-> tool_call（调用exec执行free -h命令）-> tool_result返回结果 -> assistant总结回复用户。展示token消耗：第一轮14032输入token + 294输出token，约0.1美元。解释为何简单问题也消耗大量token——因为系统提示词在首轮对话就全部注入。

### 3. 复杂任务实测：生成AI新闻PDF（02:26 - 03:51）
使用MiniMax 2.1模型（可展示思考过程），要求"把今日的AI新闻整理成PDF发给我"。后台进行了约70多次对话：web search搜索新闻 -> markdown转PDF过程中遇到困难 -> 安装各种依赖 -> 费了九牛二虎之力完成PDF -> 飞书发送又出问题。最终markdown文件尚可，PDF效果一般。

### 4. 定时任务实测：监控推特（03:51 - 04:31）
测试OpenClaw独有的定时任务功能——监控某人推特并即时通知。后台实现原理：增加一个cron配置文件，定时将提示词发给Agent。检查状态文件发现时间根本没找对，后续工作也无需检查。后台日志显示Agent"像个执着而倔强的实习生"，接口不通、返回值不符预期、安装命令失败等问题层出不穷。

### 5. Agent本质问题的深刻比喻（04:31 - 06:42）
通过一个夸张但精准的"老板问今天几号"比喻，揭示当前Agent的核心问题：虽然每一步思考都合理（遇到问题->解决问题），但整体行为链条荒谬——手机没电->找充电器->充电器没有->去超市买->超市关门->打车去另一家->手机没电付不了款->去银行取钱->钱不够->抵押手机给司机->买完充电器发现手机丢了->贷款买新手机。最终花了一整天、丢了旧手机、欠了贷款，才回答"今天是2月6号"。指出Agent应主动反馈和商量，而非一味蛮干。而且这个Agent还要按自己付出的努力来收费。

## 核心概念清单

| 概念 | 解释 |
|------|------|
| OpenClaw (open cloud) | 一个可部署在本地或云端的Agent框架，最大特点是可接入社交媒体触发任务 |
| Agent | AI智能体，能够自主调用工具、执行多步骤任务的AI系统 |
| 飞书机器人 | 飞书开放平台上创建的机器人，作为用户与Agent交互的通道 |
| OpenRouter | 大模型API聚合平台，提供多种模型的统一调用接口 |
| token | 大模型处理文本的基本单位，输入和输出token的数量直接影响费用 |
| role字段 | 对话中的角色标识：user（用户）、assistant（大模型）、tool_result（工具返回结果） |
| tool_call | Agent请求调用外部工具的机制，格式包含方法名和参数 |
| exec | OpenClaw内置工具，用于在服务器上执行shell命令 |
| 系统提示词 (System Prompt) | Agent启动时注入的大量指令文本，定义了Agent的能力和行为规范 |
| cache | 缓存机制，用于减少重复的token消耗 |
| MiniMax 2.1 | 国产大模型，支持展示思考过程 |
| cron定时任务 | 通过配置文件定时触发Agent执行任务的机制 |
| web search | Agent内置的网络搜索工具 |
| RSS | Really Simple Syndication，一种信息聚合协议，视频中提到RSS.app作为更合理的监控方案 |
| API Key | 用于身份验证和API访问的密钥 |
| app id / secret key | 飞书机器人的身份凭证 |
