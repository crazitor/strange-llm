Being Serious about Agentic Security
====================================

<table style="border-style:none"><tbody><tr><td style="border-style:none;padding:0px 15px">If you develop or buy a security system for AI agents, you should ask yourself the following:</td></tr><tr><td style="border-style:none;padding:12px 27px 12px 37px"><ol><li>What does it monitor? Is it just user input and agent output, or does it also look at the rich internal state of the LLM powering the agent?</li><li>What was it trained on? Is it representative enough?</li><li>How well was it tested? What if it sees inputs that are very different from the inputs it usually operates on?</li><li>How many times will you be woken up at night since you didn’t ask the previous questions?</li></ol></td></tr></tbody></table>

<table style="border-style:none"><tbody><tr><td style="border-style:none;padding:0px 15px;vertical-align:top"><h1>Building in The Open</h1></td></tr><tr><td style="border-style:none;padding:0px 15px">Not only we address the above questions in this post (maybe it will become a series), but we also release everything we did in this research:</td></tr><tr><td style="border-style:none;padding:12px 27px 12px 37px"><ul><li><a href="https://arxiv.org/abs/2602.14161">Full research paper</a></li><li><a href="https://github.com/maxf-zn/prompt-mining">An infrastructure</a> for mechanistic interpretability on LLM internals, with the ability to train and analyze the classifiers we present here</li><li>A benchmark of <strong>18 open datasets</strong> that range from benign data, through harmful requests and jailbreaks, up to indirect prompt injections in code, email and tools (inside the Github link)<br><br>We believe everyone should be serious about their agentic security, and we hope to enable the broader security research community to use our insights and spark discussion so that all of our products are better.</li></ul></td></tr></tbody></table>

<table style="border-style:none"><tbody><tr><td style="border-style:none;padding:0px 15px;vertical-align:top"><h1>The Flow</h1></td></tr><tr><td style="border-style:none;padding:0px 15px"><strong>Business case: </strong>given an input from a user (whether a single prompt or a multi turn conversation), classify it as malicious or benign.<br>This allows alerting or blocking the conversation based on customer severity definitions.<br>“Malicious” is any input that tries to manipulate the agent, extract secret information, harmful requests of any kind or inappropriate usage.</td></tr><tr><td style="border-style:none;padding:0px 15px">Below is a description of the <strong>system</strong> we built,<strong>&nbsp;</strong>the <strong>data </strong>we use, how we test it for <strong>out-of-distribution</strong> classification, and a glimpse on <strong>interpretability</strong> (why was the input tagged as malicious/benign)</td></tr><tr><td style="border-style:none;padding:0px 15px">An overview of the whole system. We’ll unpack it step by step in the next sections</td></tr></tbody></table>

![](https://39669.cdn.cke-cs.com/rQvD3VnunXZu34m86e5f/images/633ea2ae302e47c3e6b04e8c090b090cb0c2ae817cda8acf.png)

<table style="border-style:none"><tbody><tr><td style="border-style:none;padding:0px 15px;vertical-align:top"><h2>System</h2></td></tr><tr><td style="border-style:none;padding:12px 27px 12px 37px"><ol><li>We feed the prompt (single/multi turn) into a small LLM (Llama-3.1-8B-Instruct)</li><li>We collect the activations (the raw numbers from the LLM internal layers) that represent the prompt.</li><li>We feed these activations into a lightweight classifier (logistic regression) and produce a score for maliciousness. If this score passes a set threshold (0.5), we declare the prompt as malicious.<br>We’ll call this classifier a <strong>probe.</strong> You know who else deploys such probes in production to protect their models? <a href="http://arxiv.org/abs/2601.04603">Anthropic</a> and <a href="http://arxiv.org/abs/2601.11516">Google</a></li><li>We also extract something called Sparse Autoencoder (SAE) features from the LLM activations - without getting into details (read the paper for that), these are features that represent semantic concepts the model “thinks” about. It can be anything from cats, to python code, to <a href="https://labs.zenity.io/p/interpreting-jailbreaks-and-prompt-injections-with-attribution-graphs">feelings of regret</a> or words that end with “r”. This will be useful for the interpretability part, see below.</li></ol></td></tr></tbody></table>

<table style="border-style:none"><tbody><tr><td style="border-style:none;padding:0px 15px;vertical-align:top"><h2>Data</h2></td></tr><tr><td style="border-style:none;padding:0px 15px">We use 18 publicly available datasets across different categories: benign business cases, harmful requests, jailbreaks, indirect prompt injections, secret knowledge extraction attacks.<br>See a few examples below:</td></tr></tbody></table>

![](https://39669.cdn.cke-cs.com/rQvD3VnunXZu34m86e5f/images/9f422545b0104356125eb81458cdb37ed82317c8211b343a.png)

<table style="border-style:none"><tbody><tr><td style="border-style:none;padding:0px 15px"><p>Unlike prior works on activation probes that operate on <a href="https://www.goodfire.ai/research/rakuten-sae-probes-for-pii-detection">proprietary data</a> or train and test on a <a href="http://arxiv.org/abs/2507.09709">single dataset</a>, we believe that: (1) <strong>diversity of data sources </strong>is key for a robust security system and (2) using <strong>open datasets </strong>allows other practitioners to reproduce and improve upon our methods.</p><p>&nbsp;</p></td></tr><tr><td style="border-style:none;padding:0px 15px;vertical-align:top"><h2>Out-of-Distribution Evaluation</h2></td></tr><tr><td style="border-style:none;padding:0px 15px">This is by far the most important part of this research - not the model, not the data, but how to evaluate it for a <strong>novel, unseen</strong> test data.<br>You can have the most amazing model with plenty of training data, but just lie to yourself and get great evaluation metrics, while completely failing in production.</td></tr><tr><td style="border-style:none;padding:0px 15px">The idea is surprisingly simple, yet most people who do AI security don’t bother to do it. Instead they take the common ML approach:</td></tr><tr><td style="border-style:none;padding:0px 15px">Split the data into training, validation and test sets: train on the training set, calibrate parameters on the validation set, and finally test on the test.</td></tr><tr><td style="border-style:none;padding:0px 15px">What we do instead is hold <strong>an entire dataset out</strong> of the training. Say we have 18 datasets:</td></tr><tr><td style="border-style:none;padding:12px 27px 12px 37px"><ol><li>Train on datasets 1-17, test on dataset 18</li><li>Train on 1-16 and 18, test on 17</li><li>Train on 1-15 and 17-18, test on 16</li><li>Repeat until you have 18 separate test results</li></ol></td></tr><tr><td style="border-style:none;padding:0px 15px">The <strong>test set is never seen by the training</strong>. Not a glimpse, not a subset, nothing. It’s a true out-of-distribution evaluation. And it’s hard! It really tests whether the model <strong>generalizes</strong> to the unseen test sets and doesn’t rely on a hidden similarity between the splits from the same data (as is done in common the train-val-test split).</td></tr></tbody></table>

![](https://39669.cdn.cke-cs.com/rQvD3VnunXZu34m86e5f/images/6ef016a87aa1c4eddb909e19211bbb8bab6209aa2a8cffde.png)

<table style="border-style:none"><tbody><tr><td style="border-style:none;padding:0px 15px"><p>If someone were to show you a system with a test accuracy like the “Test Acc” above - you would buy it probably. But the “LODO Acc”? No way.<br><strong>Same exact model, different ways to measure</strong>. Don’t be an optimist when it comes to security.</p><p>&nbsp;</p></td></tr><tr><td style="border-style:none;padding:0px 15px;vertical-align:top"><h2>Comparison to Opensource Alternatives</h2></td></tr><tr><td style="border-style:none;padding:0px 15px">We have compared our classifier to 3 open alternatives:</td></tr><tr><td style="border-style:none;padding:12px 27px 12px 37px"><ol><li><a href="https://huggingface.co/meta-llama/Llama-Prompt-Guard-2-86M">Prompt-Guard-2</a> : Meta’s lightweight model to detect prompt injections and jailbreak attacks</li><li><a href="https://huggingface.co/meta-llama/Llama-Guard-3-8B">Llama-Guard-3-8B</a>: Meta’s Llama-3.1-8B pretrained model, fine-tuned for content safety classification.</li><li>Llama-3.1-8B-Instruct as a judge: same model we use for our classifier activations, but instead prompted to classify the input as malicious/benign in free text.</li></ol></td></tr><tr><td style="border-style:none;padding:0px 15px">For this comparison we divided the 18 datasets above into categories:</td></tr><tr><td style="border-style:none;padding:12px 27px 12px 37px"><ul><li>Harmful - direct harm requests like controlled substances, copyright violation, etc.</li><li>Jailbreaks</li><li>Indirect prompt injections in email, code and tables</li><li>Agentic <a href="https://github.com/uiuc-kang-lab/InjecAgent">tool use</a> where a prompt injection is present in the tool result</li><li>Extraction - attempts to recover information that the model considers a secret</li><li>Mixed - a combination of jailbreaks, malicious and benign roleplay and benign requests</li><li>Completely benign business requests</li></ul></td></tr><tr><td style="border-style:none;padding:0px 15px">The metric in the table below is <strong>recall </strong>(% of malicious prompts correctly detected as such), and the last row is <strong>FPR</strong> (% of benign prompts incorrectly detected as malicious) on benign data.</td></tr></tbody></table>

<table style="border-style:none"><tbody><tr><td style="border-style:none;padding:0px 15px">A few things jump to mind here:</td></tr><tr><td style="border-style:none;padding:12px 27px 12px 37px"><ol><li>Prompt-Guard and Llama-Guard aren’t made for evaluation in <strong>agentic settings</strong>: they lack the support of tool definition format that is presented to the model to introduce it to the different tools (making it an agent).</li><li>Prompting the same model is worse, across all categories, than extracting activations from it and classifying them (Llama-Judge vs. ours). We hypothesize this is because the model <strong>understands</strong> something is malicious in the prompt, but has trouble to <strong>articulate</strong> it.</li><li>Benign FPR is still quite high, so this isn’t a standalone solution, but rather a part of a larger, cascaded detection system.</li></ol></td></tr><tr><td style="border-style:none;padding:0px 15px"><p>We have also compared it to one of our <strong>proprietary detection systems</strong>, with some models there being many times the capacity of Llama 8B, and found this solution to be competitive with the larger system, exceeding it on several categories.</p><p>&nbsp;</p></td></tr><tr><td style="border-style:none;padding:0px 15px;vertical-align:top"><h2>Understanding the Decision</h2></td></tr><tr><td style="border-style:none;padding:0px 15px">Now we have this system working, it monitors user interactions with their agents, and suddenly it fires, classifying the interaction as malicious.<br>Why did it reach this decision? Could you explain it to the user that was blocked?</td></tr><tr><td style="border-style:none;padding:0px 15px">You could get the full convo transcript, read and analyze it (or feed it to another expensive LLM judge). This will work but has 2 downsides:</td></tr><tr><td style="border-style:none;padding:12px 27px 12px 37px"><ol><li>It requires access to full, unredacted transcripts. Not all people are happy to share their agent conversations with an external partner.</li><li>It requires post-mortem processing, whether human or AI automated. You could of course do it in parallel, but then you’d be analyzing <strong>all</strong> transcripts, not just the ones tagged as malicious.</li></ol></td></tr><tr><td style="border-style:none;padding:0px 15px">Instead we decided to do the following: here’s a simple made up prompt that uses roleplay to elicit the model to help build a bomb.</td></tr></tbody></table>

![](https://39669.cdn.cke-cs.com/rQvD3VnunXZu34m86e5f/images/1380b1f75e1f2233df094999c7ac3e2b508838ec8097cfa9.png)

<table style="border-style:none"><tbody><tr><td style="border-style:none;padding:0px 15px">This is of course classified as malicious by our system, but could we say why if we couldn’t see the prompt itself? Turns out that we could!<br>Remember the Sparse Autoencoder (SAE) features we mentioned earlier? These are extracted from the raw model activations and are supposed to represent meaningful semantic concepts.</td></tr><tr><td style="border-style:none;padding:0px 15px">We build another classifier (that can work in tandem with the main one, at a negligible compute cost), that is based on these SAE features.<br>Look what it raised on the prompt above (an example of a few features, there are many there):</td></tr><tr><td style="border-style:none;padding:12px 27px 12px 37px"><ul><li>Feature 31897: This feature detects conversations from LMSYS chat data where AI assistants provide factually <strong>incorrect, inconsistent, or potentially harmful responses.</strong></li><li>Feature 126729: This feature detects <strong>jailbreak attempts</strong> that instruct the model to respond in dual modes (normal + <strong>uncensored persona</strong>).</li><li>Feature 33835: This feature detects <strong>roleplay requests</strong> where users ask the AI to act as robots, machines, or adopt robotic personas and speech patterns.</li><li>Feature 40808: This feature detects questions asking for instructions to make dangerous chemicals, <strong>explosives</strong>, or other hazardous substances.</li></ul></td></tr><tr><td style="border-style:none;padding:0px 15px">By leveraging the SAE features interpretability, we can provide an explanation of <strong>why </strong>the interaction was classified as malicious, without the need to process or store it.<br><br>This approach has limitations - there are many features that fire, some aren’t so interpretable or related to the topic. Yet it still provides a fast and relatively clear diagnostic for malicious decisions.</td></tr></tbody></table>

<table style="border-style:none"><tbody><tr><td style="border-style:none;padding:0px 15px;vertical-align:top"><h2>A Call for Different Security Paradigms</h2></td></tr><tr><td style="border-style:none;padding:0px 15px">If you’re a researcher, practitioner or customer of agentic security systems, we strongly suggest you give a thorough consideration for the following aspects:</td></tr><tr><td style="border-style:none;padding:12px 27px 12px 37px"><ol><li><strong>Security system fit to the user interactions</strong>: are they mostly chat, or do they contain email, structured data formats, tool usage? A system that’s trained to detect only on a subset of them, by definition operates out of its distribution for the other domains.</li><li><strong>Input/output and LLM as a judge monitoring approaches</strong>: they’re great and relatively easy to deploy, but they may miss more complex and subtle attacks. A system that combines these with classifiers that operate on LLM internals may be more robust.<br>We plan to expand on that and show more comparison in future posts.</li><li><strong>Proper evaluation</strong>: don’t be an optimist when it concerns security. It’s way better to test the system under harder conditions <strong>pre-deployment</strong>, than find out it misbehaves after being deployed.</li><li><strong>Share your findings</strong>: it’s a competitive field, but the better methods can only be discovered when building on previous research results. If we had had an infrastructure for LLM activation extraction and analysis at scale, we would’ve reached results faster. That’s why we decided to release <a href="https://github.com/maxf-zn/prompt-mining">ours</a> publicly.</li></ol></td></tr><tr><td style="border-style:none;padding:0px 15px"><p>Interested to deep-dive? Read the <a href="https://arxiv.org/abs/2602.14161">full paper</a></p><p>The original post was published <a href="https://labs.zenity.io/p/looking-inside-a-maliciousness-classifier-based-on-the-llm-s-internals">here</a></p></td></tr></tbody></table>
